# phala Provider

Deploy confidential applications on Phala Cloud with Terraform.

This provider is built around an app-first workflow:

- `phala_app` manages one application identity with shared compose, shared environment, and one or more CVM replicas
- `phala_cvm_power` controls runtime power state for an existing CVM
- `phala_ssh_key` manages reusable SSH public keys in your Phala Cloud account

## Quick Start

```terraform
terraform {
  required_providers {
    phala = {
      source = "phala-network/phala"
    }
  }
}

provider "phala" {}

resource "phala_app" "hello" {
  name      = "hello-phala"
  size      = "tdx.medium"
  region    = "US-WEST-1"
  image     = "dstack-dev-0.5.7-9b6a5239"
  disk_size = 40
  replicas  = 1

  docker_compose = <<-YAML
    services:
      web:
        image: nginx:stable
        ports:
          - "80:80"
  YAML

  wait_for_ready       = true
  wait_timeout_seconds = 900
}

output "app_id" {
  value = phala_app.hello.app_id
}

output "endpoint" {
  value = phala_app.hello.endpoint
}
```

## Authentication

Set your API key in the environment before running Terraform:

```bash
export PHALA_CLOUD_API_KEY="phak_xxx"
```

The provider uses the public Phala Cloud API by default. Most users do not need to set `api_prefix`.

## Common Workflow

1. Use data sources such as `phala_images`, `phala_sizes`, and `phala_regions` to discover valid inputs.
2. Define one `phala_app` with your Docker Compose file.
3. Wait for readiness and consume `app_id`, `primary_cvm_id`, `cvm_ids`, and `endpoint` as outputs.
4. Use `phala_cvm_power` when you need explicit start or stop control after deployment.
